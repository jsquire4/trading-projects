BLUF — What Meridian Is in 60 Seconds

Meridian is a same-day binary stock outcome market on Solana. Pick a stock, pick a strike price, bet Yes or No on whether it closes at or above that strike by 4 PM ET. Winners get $1 per contract. Losers get nothing.

How It Works

  1. Morning: Markets are auto-created for MAG7 stocks with vol-based strike ladders.
  2. During the day: Users trade Yes/No contracts through a single on-chain order book. An AMM bot seeds liquidity.
  3. 4 PM ET: Oracle writes the closing price on-chain. The program settles every market, cranks out resting orders, auto-redeems winners, closes accounts, and creates tomorrow's markets.

Key Design Choices

  • One order book, three sides: Buy Yes (USDC bid), Sell Yes (Yes ask), Sell No (No-backed bid). Buy No is synthesized as mint_pair + sell_yes.
  • $1 invariant: Every Yes + No pair is backed by exactly $1 USDC. When a Yes seller meets a No seller, the pair burns and releases $1.
  • Fully autonomous lifecycle: Settlement, redemption, cleanup, and next-day creation are all handled by one service with no manual intervention.
  • Frontend talks directly to the chain: No backend API for trading. The wallet builds versioned transactions with ALT compression.

Full Architecture Summary

This document is a code-grounded architecture summary of the Meridian monorepo after a fresh project reindex. Meridian is a Solana-based 0DTE binary stock outcome market system: each market asks whether a stock closes at or above a strike that day, users trade Yes and No exposure through one on-chain order book, and markets settle against an on-chain oracle-fed close price with automated cleanup, redemption, and next-day market creation.

The most important architectural choice is the single-book, three-side market model. Rather than maintaining separate Yes and No books, Meridian stores one on-chain order book and represents all user-facing flows through three raw sides: USDC bid for Buy Yes, Yes ask for Sell Yes, and No-backed bid for Sell No. Buy No is synthesized client-side as an atomic mint_pair + sell_yes flow.

1. Executive Summary

Meridian is best understood as three cooperating planes. The on-chain plane is authoritative: the meridian program owns market state, token mints, vaults, escrows, settlement flags, fees, pause state, and the sparse custom order book. The off-chain operations plane keeps the protocol alive: services feed stock prices on-chain, initialize fresh daily strike markets, settle expired markets, index events, seed liquidity, and monitor health. The frontend plane is a direct wallet-first client: it reads chain state directly, constructs versioned Solana transactions itself, and uses backend HTTP only for analytics, proxies, and convenience routes.

The protocol is not a traditional order book with a separate No market. It uses a unified market topology where Yes and No are complements around a hard $1 invariant. A Yes + No pair is always backed by exactly $1 USDC in the market vault. That invariant makes the merge/burn path possible: when a Yes seller and a No seller meet, the engine can burn one Yes and one No and release the underlying dollar from the vault, splitting it between the two sellers according to execution price.

Key architectural conclusion: the order book is not just a matching surface. It is tightly coupled to escrow routing, vault accounting, mint supply conservation, settlement automation, and the frontend’s user-intent abstraction. Most of Meridian’s complexity comes from preserving those invariants while still making all four trading actions feel simple.

3. Critical Systems

Frontend Trading Client

The frontend is not a thin shell over an API. It derives PDAs, reads Anchor accounts, deserializes raw order-book bytes, loads ALTs, and builds versioned transactions directly against the program. This makes wallet UX first-class and keeps execution trust-minimized.

Key files: app/meridian-web/src/hooks/useAnchorProgram.ts, app/meridian-web/src/hooks/useMarkets.ts, app/meridian-web/src/hooks/usePlaceOrder.ts, app/meridian-web/src/lib/orderbook.ts

Meridian On-Chain Program

The meridian program owns the market lifecycle from config initialization to pair minting, order matching, fee charging, settlement, redemption, crank cleanup, pause controls, and closure. Every economically meaningful state transition is encoded here.

Key file: programs/meridian/src/lib.rs

Mock Oracle Program

The oracle program stores price feeds as on-chain accounts. Off-chain feeders write prices in, and the normal settle_market path reads them back on-chain. The overall settlement pipeline can still fall back to admin-mediated settlement from last-known external prices if confirmation times out, so the oracle-backed path is the primary path rather than the only path.

Key files: programs/mock-oracle/src/lib.rs, programs/mock-oracle/src/state/price_feed.rs

Settlement Pipeline

The settlement service is the operational heart of the system. It detects expired unsettled markets, confirms prices, updates the oracle, settles, crank-cancels resting orders, auto-redeems, closes old markets, creates next-day markets, and unpauses when appropriate.

Key files: services/settlement/src/index.ts, services/settlement/src/settler.ts, services/settlement/src/cranker.ts, services/settlement/src/redeemer.ts, services/settlement/src/closer.ts

Market Initialization

Daily markets are created per ticker and strike with full PDA wiring, mints, vaults, escrows, and an Address Lookup Table. This is what turns a stock ticker and strike ladder into tradeable on-chain markets.

Key files: services/market-initializer/src/index.ts, services/market-initializer/src/initializer.ts

Analytics and Event Indexing

Trading itself does not require a backend API, but portfolio history and analytics do. The event indexer listens to Anchor logs, decodes emitted events, stores them in SQLite, and serves REST endpoints for fills, VWAPs, cost basis, and portfolio snapshots.

Key files: services/event-indexer/src/listener.ts, services/event-indexer/src/db.ts, services/event-indexer/src/api.ts

5. Order Book Core Model

Meridian keeps one custom sparse order book account per strike market. The account starts with a compact header and allocates price levels and slots on demand. This replaced a dense fixed-size layout and dramatically reduced creation cost. The order book is sparse but still deterministic: it has a price map, per-level headers, and order slots containing owner, order id, remaining quantity, side, timestamp, active flag, and rent depositor.

Why a Single Book

A separate Yes book and No book would double storage, double matching complexity, and require cross-book arbitrage logic. Meridian instead expresses the No side through complementary pricing and a No-backed collateral path inside the same book.

Why Sparse Allocation

Most markets are lightly populated. Sparse allocation means markets start small and grow only when a new price level or slot is needed. This materially lowers rent and makes permissionless market creation feasible.

Matching Policy

Matching is strict price-time priority. Incoming bids scan asks from low to high, incoming asks scan bids from high to low, and within a price level the oldest order wins. When a Yes ask sees both USDC bids and No-backed bids at the same price, timestamp order decides; there is no bias toward one bid type.

Settlement Paths

There are only two fill settlement paths: a standard swap for USDC bid x Yes ask, and a merge/burn for No-backed bid x Yes ask. Everything else in the trade engine ultimately reduces to those two branches.

Important nuance: the engine itself is pure matching logic over raw order-book bytes. Token transfers, burns, fee application, and vault accounting happen after the engine returns fills. That separation keeps matching deterministic and lets the instruction layer handle asset movement safely.

6. The Three Sides

Meridian exposes four user-visible actions but stores only three raw on-chain order sides. This is the foundation of the entire trading model.

Raw Side User Intent Collateral Escrowed Price Meaning Can Match With Settlement Branch
0 = USDC bid Buy Yes USDC in escrow_vault Direct Yes price in cents Yes ask Standard swap
1 = Yes ask Sell Yes Yes in yes_escrow Direct Yes sale price in cents USDC bid and No-backed bid Swap or merge/burn
2 = No-backed bid Sell No No in no_escrow Stored in Yes-price terms
sell_no(40c) => bid 60		 Yes ask Merge/burn

Side 0: Buy Yes

This is a normal dollar-backed bid. The user escrows price-adjusted USDC and competes for Yes asks. When filled, the buyer receives Yes and the seller receives USDC from the order escrow. The main vault is untouched because the trade is a secondary-market swap, not a redemption.

Side 1: Sell Yes

This is the bridge side. A Yes seller can either sell to a bullish USDC-backed buyer or meet a No seller and trigger merge/burn. Side 1 is therefore the only side that interacts with both other raw sides.

Side 2: Sell No

This is the architectural trick. A No seller is represented as a bid because they are offering complementary value against Yes asks. Their No token is valid collateral because, when paired with a Yes, it releases exactly $1 from the vault.

Buy No is not a native raw side. The frontend synthesizes it as: mint_pair(quantity) then place_order(side=1, complementary_yes_price), followed in the current frontend implementation by a cleanup redeem(mode=0) for market orders. Economically, the user spends $1, sells the Yes leg, and keeps the No leg at a net cost of 1 - yes_sale_price.

7.1 Step-by-Step Lifecycle

  1. The frontend loads the selected market, order book snapshot, and any market ALT so it can build a versioned transaction.
  2. The user’s action is translated to a raw on-chain representation. Buy No becomes a composed multi-instruction transaction.
  3. The program validates market status, side, price bounds, quantity, and position conflict rules.
  4. Collateral is moved into the correct escrow account before matching.
  5. The matching engine scans the opposite side using price-time priority and returns fills up to max_fills.
  6. Each fill is processed as either a standard swap or a merge/burn.
  7. Events are emitted, maker slots are decremented or deactivated, and leftover quantity either rests or refunds.
  8. Off-chain indexing turns emitted events into portfolio history, cost basis, VWAP, and trade feed data.

7.2 The Full Order Flow for Each of the Three Raw Sides

Side 0 Flow: Buy Yes

The wallet submits place_order(0, price, qty). USDC is escrowed into escrow_vault. The engine scans Yes asks from cheapest upward. If matched, the Yes seller receives USDC from escrow and the buyer receives Yes. If partially matched and the order is a limit order, the unfilled remainder rests on-book backed by the remaining escrow. If it is a market order, unused USDC returns to the user.

Side 1 Flow: Sell Yes

The wallet submits place_order(1, price, qty) or the frontend synthesizes Buy No through this side. Yes moves into yes_escrow. The engine scans the combined bid world: direct USDC bids and No-backed bids. Against a USDC bid, the fill is a normal trade. Against a No-backed bid, the fill becomes merge/burn and the main vault releases the dollar locked behind the pair.

Side 2 Flow: Sell No

The wallet submits place_order(2, inverted_yes_price, qty). No moves into no_escrow. The engine scans Yes asks up to the complementary max price. On fill, the program burns the matched Yes and No tokens and releases exactly $1 per pair from usdc_vault, splitting proceeds between the Yes seller and the No seller.

7.3 Buy No, Sell No, and Cross-Seller Completion

The deepest part of the design is that three-sided matching completes the full user trade space. Buy Yes and Sell Yes are obvious. Sell No becomes native through side 2. Buy No is then achieved by minting a complete pair and immediately selling the Yes leg into the book. This yields all four user actions without needing a second order book.

It also allows a non-obvious but important emergent behavior: a Yes seller and a No seller can complete a trade with no external buyer. Their complementary positions meet inside the matching engine, burn against each other, and unlock the dollar in the vault. This is why the order book is not merely “a Yes-vs-USDC CLOB”; it is really a complete-set netting engine expressed through a CLOB interface.

8. Blockchain State and Invariants

Core Accounts

  • GlobalConfig: global admin, fees, thresholds, treasuries, registry references, pause state.
  • TickerRegistry: supported tickers and active/inactive state.
  • StrikeMarket: one market per ticker/strike/day with vault and escrow addresses, oracle feed, settlement flags, counters, and ALT.
  • OrderBook: sparse raw book storage for active orders.
  • PriceFeed: on-chain oracle record per ticker.

Market Asset Topology

  • yes_mint and no_mint: complementary outcome tokens.
  • usdc_vault: mint collateral backing all unreedeemed pairs.
  • escrow_vault: active USDC collateral for side 0 bids.
  • yes_escrow: active Yes collateral for side 1 asks.
  • no_escrow: active No collateral for side 2 bids.
Invariant Meaning Why It Matters
usdc_vault = (total_minted - total_redeemed) * $1 Every live Yes+No pair has one dollar behind it. Guarantees redemption and merge/burn payouts remain solvent.
yes_supply = no_supply Complementary token supply always stays equal. Prevents one-sided token creation or destructive accounting drift.
Escrow balances equal active book obligations Each escrow account reflects exactly the collateral behind open orders of that side. Ensures cancel and fill logic can always return or settle assets correctly.
Merge/burn payout conservation Yes seller payout + No seller payout = $1 per burned pair. Preserves the complete-set identity during direct seller-to-seller completion.
The vault invariant is the protocol’s economic anchor. Trading escrows are deliberately separated from the main vault so normal order-book activity never muddies the accounting of minted collateral versus redeemed collateral.

9.6 Position Constraints

ConflictingPosition: The current program blocks the main trading entry points from entering a conflicting wallet state, but it does not make the broader statement "holding both is impossible" universally true. The atomic Buy No path is transient, while standalone mint_pair can still mint both tokens. The enforced checks are:
  • Side 0 (Buy Yes): requires user_no_ata.balance == 0
  • Side 2 (Sell No): requires user_yes_ata.balance == 0
  • mint_pair: requires user_yes_ata.balance == 0

  • 6159: UnsettledMarketsExist

Sparse Book (6170-6175)

  • 6172: InsufficientRentDeposit
  • 6173: MaxLevelsReached

9.9 Override Window Mechanics

After settlement, the admin has a short override window to correct a bad outcome. The deadline is the maximum of two formulas to guarantee it is always in the future: 

override_deadline = max(
  clock.unix_timestamp + override_window_secs,    // always in the future
  settled_at + override_window_secs × (count + 2) // caps total extension
)

After 1st override:  max(now + window, settled_at + 2 × window)
After 2nd override:  max(now + window, settled_at + 3 × window)
After 3rd override:  max(now + window, settled_at + 4 × window) (then blocked: MaxOverridesExceeded)

override_window_secs: configurable via update_config (default 1s, range 1–3600s)
Redemptions blocked until: clock.unix_timestamp >= override_deadline
Crank redeem blocked until: same condition

9.10 Crank Redeem Delegation

How auto-redemption works: During mint_pair, the market PDA is set as a delegate on both the user's Yes and No ATAs. This allows crank_redeem to burn winning tokens on behalf of users without their signature. The instruction validates that delegated_amount ≥ balance and the delegate matches the market PDA. Users can always call redeem manually without delegation.

11. Fee and Treasury Architecture

Meridian uses a three-vault financial architecture. Understanding which vault holds what, and the accounting rules governing each, is critical for reasoning about solvency, withdrawal safety, and SOL rent management.

USDC Treasury

Seeds: ["treasury"]

Holds escrow dust swept during close_market. Withdrawals are hard-capped: available = balance - config.obligations. The obligations field tracks total USDC owed to winning token holders across all settled-but-not-yet-redeemed markets. This prevents the admin from withdrawing funds that users are entitled to.

Fee Vault

Seeds: ["fee_vault"]

Holds accumulated protocol trading fees. Fully withdrawable by admin via withdraw_fees with no cap or constraint. Fees are deposited per-fill during place_order.

SOL Treasury

Seeds: ["sol_treasury"]

System account (0 data bytes) that holds SOL for market creation rent reimbursement. Receives rent from closed token accounts, order books, and market PDAs via close_market. Withdrawal floor: rent_exempt_minimum + operating_reserve.

Fee Calculation

Per fill during place_order:
  fee = floor(gross_usdc × config.fee_bps / 10,000)

In swap fills:   fee deducted from USDC transferred to maker (net = gross - fee)
In merge fills:  fee deducted from $1 gross payout before the Yes/No split
Fee transferred to fee_vault (config PDA signs)

Max fee_bps: 1000 (10%). Set via update_fee_bps (admin-only).

Strike Creation Fee

Non-admin market creators pay config.strike_creation_fee in USDC, transferred from their USDC ATA to the fee vault. Admin-created markets are exempt. The fee is capped at 1,000 USDC (1,000,000,000 lamports) to prevent accidental bricking. Set via update_strike_creation_fee.

Market Creation Reimbursement

When the admin creates a market, the SOL rent for all 8 accounts (market PDA, 2 mints, 4 token accounts, order book) is reimbursed from the SOL Treasury if funds are available: 

available = sol_treasury.lamports - operating_reserve - rent_exempt_min
reimburse = min(total_rent, available)
Transfer: SOL Treasury → admin wallet

Obligations Tracking

When a market settles, config.obligations increases by total_minted - total_redeemed (the outstanding winning token supply). Both winner redemptions (redeem mode 1, crank_redeem) and pair burns (redeem mode 0) decrement obligations via saturating_sub. This counter ensures the treasury can never be drained below what users are owed.

12. Automations and Daily Operations

Oracle Feeder

Polls or synthesizes market prices, updates the on-chain oracle, and can trip the circuit breaker if market-state anomalies appear. It is effectively the real-time bridge from stock price observation into chain-readable settlement data.

Market Initializer

Builds each day’s markets by deriving strikes, market-close timestamps, PDAs, and ALTs, then calling create_strike_market. This is the factory for fresh tradeable contracts.

Settlement Service

Runs a reactive poller rather than a naive fixed cron. It detects expired unsettled markets, confirms close prices, writes confirmed oracle values, settles, cranks, auto-redeems, closes, initializes the next session’s markets, and unpauses as needed.

Automation Scheduler

Three scheduled jobs: 8:30 AM ET morning health check (verifies markets exist, fallback creates if missing), 4:05 PM ET settlement trigger (POSTs to settlement service as belt-and-suspenders backup), and midnight timer recalculation for DST handling.

Event Indexer + Signals

Converts Anchor log events into queryable data products for history, cost basis, PnL, VWAPs, and portfolio snapshots. Also computes the Meridian Index (aggregate MAG7 sentiment, 0–100) and Time-Weighted Conviction Scores (per-wallet prediction credentialing). Snapshots the index every 5 minutes. Exposes signals via /api/index/*, /api/conviction/*, and /api/signals/smart-money.

Monitor and AMM Bot

monitor watches liveness, stale oracle conditions, overdue settlements, and admin balances. The AMM bot seeds liquidity using quantitative pricing logic so newly created markets are tradeable immediately.

All services under services/ are long-running daemons. market-initializer runs an initial creation cycle on startup, then stays alive as an HTTP trigger server (POST /trigger) for on-demand market creation by the settlement pipeline or admin UI.

13. Service Internals

Each service has specific polling intervals, pricing models, error handling strategies, and data schemas. This section documents the operational details that the high-level automation overview omits.

13.1 Oracle Feeder

Polling and Rate Limiting

  • Poll interval: 10s during market hours, 5 min when closed
  • Per-ticker rate limit: 5s minimum between on-chain updates
  • Price conversion: price × 1,000,000 (dollars → USDC lamports)
  • Confidence: price × 1,000,000 × 0.001 (0.1% of price)
  • Timestamp: current unix_timestamp at write time

Circuit Breaker Capability

  • Monitors external market state (Yahoo marketState field)
  • If market closes before market_close_unix, trips pause() on Meridian
  • Handles AlreadyPaused (6023) gracefully
  • Seeds prices on startup regardless of market state
  • Current feeder implementation reads tickers from legacy on-chain GlobalConfig.tickers (or env override), even though broader market validation has moved toward TickerRegistry

13.2 AMM Bot

Pricing Model

  • Fair value: Black-Scholes binary call N(d2)
  • Inputs: spot (oracle), strike, T (time to close in years), vol (30-day HV, cached 5 min), r (default 5%)
  • Spread: spreadBps / 10000 × fairPrice (default 500 bps = 5%)
  • Inventory skew: -(inventory / maxInventory) × skewFactor × halfSpread
  • MinEdge: 1 cent between bid and ask

Execution and Safety

  • Poll interval: 30s
  • Per-cycle: cancel all bot orders, then place fresh bid + ask
  • Default quantity: 1,000,000 lamports ($1) per side
  • Halt conditions: |inventory| > maxInventory or consecutiveErrors > 5
  • Skips if global pause active
  • Guard against overlapping polls (isPolling flag)

13.3 Event Indexer

Database Schema (SQLite)

events:
  id, type, market, data (JSON),
  signature (unique), slot, timestamp,
  seq, created_at

checkpoints:
  id=1 (singleton), last_signature,
  last_slot, updated_at

order_intents:
  order_id+market+wallet (composite PK),
  intent (buy_yes|sell_yes|buy_no|sell_no),
  display_price (1-99), created_at

REST API Endpoints

  • GET /api/events — query by market, type, limit (≤500)
  • GET /api/events/latest — latest 20 events
  • GET /api/events/cost-basis?wallet= — per-market avg price, total qty, total cost
  • GET /api/events/market-vwaps — VWAP per market
  • GET /api/events/fills?wallet= — fills with viewer-perspective intent
  • GET /api/portfolio/snapshot?wallet= — aggregated positions

  • POST /api/order-intent — store user's trade intent for labeling
Backfill: On startup, the indexer walks backward through getSignaturesForAddress (batches of 100, 10 parallel getTransaction calls) from newest to the stored checkpoint. Events are parsed from Anchor program logs via base64 Borsh decoding. The live listener runs concurrently so no events are missed during backfill.

13.4 Monitor Service

Polls every 5 minutes (configurable). Four health checks per cycle:

  • Admin SOL balance: alerts if < 0.1 SOL
  • Oracle freshness: market hours only (M-F 9:30-16:00 ET), alerts if any feed > 10 min stale
  • Unsettled expired markets: flags markets past close that haven't settled, logs hours overdue
  • Closeable markets: flags settled markets > 90 days old that haven't been closed

13.5 Settlement Service Specifics

Detection and Confirmation

  • Poll interval: 60s
  • Double-confirm prices: poll twice with 30s gap, settle once prices stabilize
  • Confirmation timeout: 30 min
  • Oracle retry for transient errors (6040 OracleStale, 6041 OracleConfidenceTooWide): up to 15 min at 30s intervals
  • Admin settle fallback: kicks in after 5 min delay (ADMIN_SETTLE_DELAY) using last-known prices

Post-Settlement Pipeline

  • Crank cancel: batches of 32 orders, loops until book empty
  • Auto-redeem: 16 users per crank_redeem batch (2 ATAs per user)
  • Close market: requires yes_supply == 0 && no_supply == 0
  • Next-day init: triggers market-initializer via HTTP POST to /trigger (5 min timeout)
  • Unpause: exponential backoff, up to 5 retries

13.6 Market Initializer

Strike Selection

  • Vol-aware mode: HV20 (20-day historical vol) → daily vol (÷ √252) → strikes at ±1σ, ±1.5σ, ±2σ
  • Baseline fallback: ±3%, ±6%, ±9% + ATM (if HV20 unavailable)
  • Rounding: $10 increment for stocks ≥ $100, $5 for cheaper
  • Deduplication and sort ascending

Market Creation Flow

  • Idempotent: skips markets already on-chain
  • Creates all PDAs (market, 2 mints, 4 token accounts, order book)
  • ALT creation: createLookupTable + extend with 11 addresses (market accounts + TOKEN_PROGRAM + SYSTEM_PROGRAM)
  • DST-aware close time: 4 PM ET (or 1 PM on early-close dates, NYSE holidays hardcoded 2025-2028)
  • set_market_alt called after ALT creation (one-time)

13.7 Automation Scheduler

Three DST-aware scheduled jobs: 8:30 AM ET morning health check (spawns verify.ts, fallback creates markets if missing), 4:05 PM ET settlement trigger (POSTs to settlement service with 10-minute timeout, handles 409 "already in progress" gracefully), and midnight timer recalculation. All timers use getNextETTime() with DST-aware offset detection via Intl.DateTimeFormat.

DST handling uses Intl.DateTimeFormat with America/New_York timezone, avoiding manual offset tables. The NYSE holiday list is hardcoded for 2025-2028 and must be updated annually.

13.8 Shared Utilities

Pricer

Black-Scholes N(d2) for binary options. normalCdf via Abramowitz & Stegun approximation (error < 1.5e-7). Edge cases: T=0 or σ=0 returns payoff directly.

Greeks

Binary Δ, Γ, θ, V computed from d2 and standard normal PDF. All return 0 for degenerate inputs (T≤0, σ≤0, S≤0).

Quoter

Two-sided quote generation with spread, inventory skew, and minEdge gap. shouldHalt() returns true if |inventory| exceeds max or consecutive errors exceed 5.

Volatility

Close-to-close log returns annualized by √252. historicalVolatility(bars, 30) for 30-day window. rollingVolatility for time series.

Alerting

createLogger(service) with info/warn/error/critical levels. Critical and error POST to ALERT_WEBHOOK_URL if configured.

Market Data

IMarketDataClient interface with Yahoo and Synthetic implementations. Factory dispatches based on MARKET_DATA_SOURCE env var.

13.9 Railway Deployment

All services deploy on Railway with NIXPACKS builder. Each service has its own railway.toml: npm install build step, npx tsx src/index.ts start command, ON_FAILURE restart (max 5 retries). The event indexer adds a healthcheck at /api/health with 120s timeout.

14. End-of-Day Settlement and Opening Markets

14.1 End-of-Day Settlement Sequence

  1. The settlement poller wakes every 60 seconds and scans all markets for market_close_unix <= now while still unsettled.
  2. It checks that the external market is actually closed. If the HTTP market-state check fails, the code proceeds cautiously but still treats on-chain time as authoritative.
  3. It double-confirms close prices by polling twice with a 30-second gap until prices stabilize or a timeout is reached.
  4. Confirmed close prices are written back on-chain through the mock_oracle program.
  5. The service calls settle_market for confirmed tickers. If confirmation timed out, it can fall back to admin_settle using last-known prices.
  6. Once markets are settled, the same service runs crank_cancel batches to refund all resting orders.
  7. It then runs auto-redeem over winning holders where appropriate.
  8. After the override window and supply checks allow it, it runs close_market to reclaim rent and sweep remaining dust paths.

14.2 Opening Markets / Next-Day Availability

There is no explicit on-chain open_market instruction. Markets are effectively open when they have been created, their close timestamp is in the future, and the platform is not paused. In the current architecture, new markets are not primarily created by a morning cron. Instead, the settlement pipeline spawns the market initializer after the previous day’s cleanup so the next day’s markets can already exist before the next trading session.

The morning automation job is therefore a safety net, not the main path. It verifies that the expected markets are present and triggers fallback creation only if the post-settlement initialization failed or was skipped.

Operationally, Meridian “opens” by preparing future-dated markets early. The next day’s markets can be live as soon as the previous day’s settlement workflow finishes and any protective pause is lifted.

The entire stack can run without external market data by setting MARKET_DATA_SOURCE=synthetic. This enables full-stack testing, demos, and development without Yahoo Finance dependencies.

GBM Price Generation

  • Geometric Brownian Motion: S_{t+dt} = S_t · exp((μ - σ²/2)·dt + σ·√dt·Z)

  • Per-symbol SeededRng instances via hashSeed(globalSeed, symbol)
  • Tick interval: ~5 seconds (dt ≈ 8.5e-7 years)

SeededRng

  • LCG (Numerical Recipes): a=1664525, c=1013904223, m=2³²
  • next() ∈ [0, 1), nextGaussian() via Box-Muller transform
  • Base prices: AAPL=190, TSLA=250, AMZN=185, MSFT=420, NVDA=130, GOOGL=175, META=500

In synthetic mode, all services run both autonomous pollers and HTTP trigger servers. 17. Operational Notes and Risks

Authoritative Docs vs Current Runtime

The docs and code are broadly aligned that settlement is the main lifecycle orchestrator, but some service descriptions still sound more scheduler-centric than the current implementation. In practice, the morning automation job is mostly verification/fallback, while settlement handles the main end-of-day and next-day transition flow.

Startup Caveat

The top-level Makefile services target currently starts oracle-feeder, amm-bot, event-indexer, automation, and monitor, but does not directly start settlement or the one-shot market-initializer. Operationally that matters because settlement owns the critical end-of-day workflow, and next-day market creation is normally triggered from that pipeline.

Oracle / Market Data Dependency

Yahoo or synthetic market data drives the feeder and settlement confirmation logic. If it is unavailable or unstable, the system eventually falls back to admin-mediated settlement. That is safer than blocking indefinitely, but it raises operator responsibility.

Override Window and Finality

The docs note a short override window in the current devnet-oriented build. That means practical finality arrives quickly, but it also reduces the time available for admins to correct a bad oracle-derived outcome before redemptions progress.

The two most operationally important things to understand are: first, the settlement service is now the true lifecycle engine; second, the order book cannot be reasoned about in isolation from redemption, pair minting, and vault accounting. Those pieces are economically one system.

Appendix: Glossary of Abbreviations

AbbreviationStands ForWhat It Means in Meridian
PDAProgram Derived AddressA deterministic Solana account address computed from seeds and a program ID. No private key exists, so only the owning program can sign for it. Every market, vault, escrow, mint, order book, treasury, and config account is a PDA.
ATAAssociated Token AccountThe canonical SPL token account for a given wallet + mint pair. Where a user's Yes, No, or USDC tokens live. Computed deterministically so any service can locate a user's balance without searching.
ALTAddress Lookup TableA Solana account that stores a list of frequently-used addresses, allowing versioned transactions to reference them by index (1 byte) instead of full pubkey (32 bytes). Each Meridian market creates an ALT with its 11 key accounts to compress transaction size.
SPLSolana Program LibraryThe standard library of Solana programs. SPL Token is the program that manages fungible token mints, accounts, and transfers. Yes/No outcome tokens and USDC are all SPL tokens.
USDCUSD CoinThe stablecoin used as collateral throughout Meridian. On devnet, a mock USDC mint is used; on mainnet, the real Circle USDC mint. 6 decimal places (1,000,000 lamports = $1).
IDLInterface Definition LanguageAnchor's auto-generated JSON schema describing a program's instructions, accounts, and types. Meridian uses IDL files for frontend Anchor program instances but builds test instructions manually from raw bytes.
AMMAutomated Market MakerThe bot service that seeds two-sided liquidity (bid + ask) on every active market using Black-Scholes pricing with inventory skew. Polls every 30 seconds.
BPSBasis Points1/100th of a percent. Used for fee rates (fee_bps max 1000 = 10%), oracle confidence bands (confidence_bps), and AMM spread width (spreadBps default 500 = 5%).
BLUFBottom Line Up FrontAn executive summary placed at the top of a document so readers get the key takeaway immediately.
0DTEZero Days to ExpirationOptions or contracts that expire the same day they are created. All Meridian markets are 0DTE: created in the morning, settled at 4 PM ET.
MAG7Magnificent SevenThe seven default tickers: AAPL, MSFT, AMZN, NVDA, GOOGL, META, TSLA. Hard-coded in the legacy GlobalConfig.tickers array and used as the baseline ticker set.
ATMAt The MoneyA strike price equal (or nearest) to the current stock price. The strike selector generates ATM and OTM strikes based on historical volatility.
OTMOut of The MoneyA strike price away from the current stock price. For Yes: strike above current price. For No: strike below current price.
ETEastern TimeThe US Eastern timezone used for all market hours. NYSE closes at 4 PM ET. DST transitions handled via Intl.DateTimeFormat with America/New_York.
GBMGeometric Brownian MotionThe stochastic model used by the synthetic price generator. Produces realistic-looking price paths with configurable drift and volatility for testing without live market data.
HVHistorical VolatilityAnnualized standard deviation of close-to-close log returns. HV20 (20-day) is used by the market initializer for vol-aware strike selection. Annualized by multiplying daily vol by √252.
LCGLinear Congruential GeneratorThe PRNG algorithm used by SeededRng in synthetic mode. Numerical Recipes parameters: a=1664525, c=1013904223, m=2³².
VWAPVolume-Weighted Average PriceThe average fill price weighted by quantity. Computed by the event indexer as sum(price × quantity) / sum(quantity) per market.
P&LProfit and LossUnrealized P&L computed as current position value (mid price × quantity) minus cost basis (from event indexer fill history).
OHLCVOpen, High, Low, Close, VolumeStandard bar format for historical price data. Fetched from Yahoo Finance (or synthetic) for price charts and volatility calculation.
CPICross-Program InvocationWhen one Solana program calls another. The meridian program uses CPI to invoke SPL Token for transfers, mints, and burns. The event listener tracks CPI depth to parse logs correctly.
FIFOFirst In, First OutThe order matching priority within a price level. At the same price, the order with the earliest timestamp fills first.
RPCRemote Procedure CallThe Solana JSON-RPC endpoint that all services and the frontend use to read chain state and submit transactions. Configurable via RPC_URL env var.
DSTDaylight Saving TimeUS clock shift that affects ET offset (UTC-5 vs UTC-4). The automation scheduler handles this via Intl.DateTimeFormat rather than hardcoded offsets.
NYSENew York Stock ExchangeThe stock exchange whose trading hours (9:30 AM – 4 PM ET) and holiday calendar determine when Meridian markets are created, priced, and settled.
RESTRepresentational State TransferThe HTTP API style used by the event indexer service. Endpoints serve fills, cost basis, VWAPs, portfolio snapshots, and health checks as JSON.
CSVComma-Separated ValuesExport format for trade history. The frontend builds RFC 4180-compliant CSV files and triggers browser download.
CUCompute UnitsSolana's measure of transaction computational cost. Tests use unique CU limits as a deduplication nonce to prevent bankrun from rejecting identical transaction structures.
SOLSolana (native token)The native cryptocurrency of the Solana blockchain. Used to pay transaction fees and rent for on-chain accounts. The SOL Treasury PDA holds SOL for market creation rent reimbursement.
LELittle-EndianByte order used for encoding integers in PDA seeds and on-chain account data. Strike prices are encoded as u64 LE, expiry days as u32 LE.
E2EEnd-to-EndThe 3-act stress test framework that exercises the full protocol lifecycle with multiple concurrent trading agents.
